(1) Data description:

There are two pieces of data: players.csv and sessions.csv. I will address these two separately.

players.csv contains 196 observations containing 7 columns:

  • experience: experience of the player (categorical)
  • subscribe: if the player subscribed to a game-related newsletter (logical)
  • hashedEmail: the hashed email of the player (character)
  • played_hours: the number of hours played of the player in total (numerical)
  • name: the name of the player (character)
  • gender: the gender of the player (character)
  • Age: the age of the player (numerical).

There exists NA in age and played_hours, the gender column contains several genders of minority, and the experience level is categorical, containing Beginner, Amateur, Regular, Veteran, Pro. It is relatively hard to exactly quantize and order these categories of experience level from the most experienced to the least experienced.

sessions.csv contains 1535 observations containing 5 columns:

  • hashedEmail: hashed emails of the player who played in the session (character)
  • start_time: formatted start time of the session (date)
  • end_time: formatted end time of the session (date)
  • original_start_time: UNIX time format start time of the session (numerical)
  • original_end_time: UNIX time format end time of the session. (numerical)

The formatted start time and end time is not read as date format in R, so some initial wrangling must be done. Additionally, the hashedEmail is not ordered and grouped, which makes it hard to distinguish each player. Some wrangling must be done to address this issue. The formatted start and end time is non-standard, which will introduce difficulties when applying as_datetime(), so special conversion using dmy_hm() function in lubridate is needed.

Problems in data

Some potential issues in the data are the method of collecting the data; some hashedEmails in players.csv does not match with any hashedEmail with sessions.csv, indicating that these players were never recorded to play in the game. The method of collecting the data is unknown, but it is highly possible that the recorded data in players.csv is voluntarily recorded, which means some information (such as age) might be distorted or falsely inputted. The experience level is difficult to quantize and largely objective, so there might be some issues with this part too.

(2) Questions:

I will address broad question #2: "We would like to know which "kinds" of players are most likely to contribute a large amount of data so that we can target those players in our recruiting efforts."

The specific question that I have formulated is that: Assuming that one could use age to predict the playtime of players, what is the influence of gender and experience to the relationship?

The data in players.csv provided all the data necessary to address this question, since each of its entries provided the experience, age, gender, and hours played of a single player. By performing a statistical analysis on the data, I could hopefully infer a pattern and determine the effect on the RMSE of the final model that predicts hours_played of a player using their age. Given the original state of the data, some wrangling must be done; genders should be simplified to "Male", "Female", "Others" for the purpose of statistical analysis, and entries containing players who played for 0 hours might need to be purged since this indicates that they've never played the game at all.

(4) Methods and Plan

Under the assumption of age is indeed related and will influence the number of hours played, I will separately analyze the results of 4 predictions:

Differentiating experience Differentiating gender
Prediction 1 NO NO
Prediction 2 NO YES
Prediction 3 YES NO
Prediction 4 YES YES

For the sessions that are differentiating either of these categorical data, I will average the RMSE of all the regression model produced by considering the data for the variables separately (i.e. for prediction 2: three models are produced to only use rows of the data with gender=Men/Women/Other). Due to the prospected non-linear relationship in the data as shown in the visualization, I will use KNN regression to prevent linear regression from producing inaccurate results. This way, I would not have to worry about linear models being inaccurate in predicting nonlinear relationships. However, since data seems to cluster, KNN regression would inevitably encounter its limitations - difficulty when treating data when large amounts of data points are clustered together, creating a bias depending on the number of neighbours chosen. Additionally, KNN cannot predict data points that are far away from the data points, which might be a future limitation.

I will use cross-validation on different K-Values for each model separately to ensure the optimal model is selected for every subset of the data. I might use a lower number of folds (i.e. 3 or 4) due to the scarcity of the amount of data. The proportion of training set compared to validation set is dependent on the number of folds. Additionally, the data would be further divided when doing predictions after prediction 1 - into subsets of data that only contains the same of the predictor (i.e. subset containing only gender). For processing the data, I will remove the rows of data that contain NA or have 0 hours played, since these data are likely to be irrelevant in prediction - they might represent players who participated in the survey for players.csv but never played.
